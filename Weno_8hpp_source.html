57 /// \ingroup LimitersGroup
58 /// \brief A compact-stencil WENO limiter for DG
59 ///
60 /// Implements the simple WENO limiter of \cite Zhong2013 and the Hermite WENO
61 /// (HWENO) limiter of \cite Zhu2016. The implementation is system-agnostic and
62 /// can act on an arbitrary set of tensors.
63 ///
64 /// #### Summary of the compact-stencil WENO algorithms:
65 //
66 /// The compact-stencil WENO limiters require communication only between
67 /// nearest-neighbor elements, but aim to preserve the full order of the DG
68 /// solution when the solution is smooth. To achieve this, full volume data is
69 /// communicated between neighbors.
70 //
71 /// For each tensor component to limit, the new solution is obtained by a
72 /// standard WENO procedure --- the new solution is a linear combination of
73 /// different polynomials, with weights chosen so that the smoother (i.e., less
74 /// oscillatory) polynomials contribute the most to the sum.
75 ///
76 /// For the simple WENO and HWENO limiters, the polynomials used are the local
77 /// DG solution as well as a "modified" solution from each neighbor element. For
78 /// the simple WENO limiter, the modified solution is obtained by simply
79 /// extrapolating the neighbor solution onto the troubled element. For the HWENO
80 /// limiter, the modified solution is obtained by a least-squares fit to the
81 /// solution across multiple neighboring elements.
82 ///
83 /// #### Notes on the SpECTRE implemention of the WENO limiters:
84 ///
85 /// There are a few differences between the limiters as implemented in SpECTRE
86 /// and as presented in the references. We list them here and discuss them
87 /// further below.
88 /// 1. The choice of basis to represent the DG solution
89 /// 2. The system-agnostic implementation
90 /// 3. The oscillation indicator
91 ///
92 /// Finally, in 4., we will discuss the geometric limitations of the
93 /// implementation (which are not a deviation from the references).
94 ///
95 /// ##### 1. The choice of basis
96 ///
97 /// SpECTRE uses a Legendre basis, rather than the polynomial basis that we
98 /// understand to be used in the references. Because the construction of the
99 /// modified neighbor solutions and the WENO sum is geometrically motivated, the
100 /// overall algorithm should work similarly. However, the precise numerics may
101 /// differ.
102 ///
103 /// ##### 2. The system-agnostic implementation
104 //
105 /// This implementation can act on an arbitrary set of tensors. To reach this
106 /// generality, our HWENO implementation uses a different troubled-cell
107 /// indicator (TCI) than the reference, which instead specializes the TCI to the
108 /// Newtonian Euler system of equations.
109 ///
110 /// This implementation uses the minmod-based TVB TCI of \cite Cockburn1999 to
111 /// identify elements that need limiting. The simple WENO implementation follows
112 /// its reference: it checks the TCI independently for each tensor component, so
113 /// that only certain tensor components may be limited. The HWENO implementation
114 /// checks the TVB TCI for all tensor components, and if any single component is
115 /// troubled, then all components of all tensors are limited.
116 ///
117 /// When the evolution system has multiple evolved variables, the recommendation
118 /// of the references is to apply the limiter to the system's characteristic
119 /// variables to reduce spurious post-limiting oscillations. In SpECTRE,
120 /// applying the limiter to the characteristic variables requires specializing
121 /// the limiter to each evolution system. The system-specific limiter can also
122 /// implement a system-specific TCI (as the HWENO reference does) to more
123 /// precisely trigger the limiter.
124 ///
125 /// ##### 3. The oscillation indicator
126 ///
127 /// We use the oscillation indicator of \cite Dumbser2007, modified for use on
128 /// the square/cube grids of SpECTRE. We favor this indicator because portions
129 /// of the work can be precomputed, leading to an oscillation measure that is
130 /// efficient to evaluate.
131 ///
132 /// ##### 4. The geometric limitations
133 ///
134 /// Does not support non-Legendre bases; this is checked in DEBUG mode. In
135 /// principle other bases could be supported, but this would require
136 /// generalizing the many internal algorithms that assume a Legendre basis.
137 ///
138 /// Does not support h- or p-refinement; this is checked always. In principle
139 /// this could be supported. The modified neighbor solution algorithm would
140 /// need to be generalized (reasonable for simple WENO, extremely tedious for
141 /// HWENO), and the sum of neighbor solutions may need to be updated as well.
142 ///
143 /// Does not support curved elements; this is not enforced. The code will run
144 /// but we make no guarantees about the results. Specifically, the limiter acts
145 /// in the `Frame::Logical` coordinates, because in these coordinates it is
146 /// straightforward to formulate the algorithm. This means the limiter can
147 /// operate on generic deformed grids --- however, some things can start to
148 /// break down, especially on strongly deformed grids:
149 /// 1. When the Jacobian (from `Frame::Logical` to `Frame::Inertial`) varies
150 /// across the element, then the limiter fails to be conservative. This is
151 /// because the integral of a tensor `u` over the element will change after
152 /// the limiter activates on `u`.
153 /// 2. When computing the modified neighbor solution for the WENO sum, the
154 /// extrapolation or fitting procedure may not properly account for the
155 /// coordinates of the source data. If the coordinate map of the neighbor
156 /// differs from that of the local element, then the logical-coordinate
157 /// representation of the neighbor data may be incorrect. This may be a
158 /// large error at Block boundaries with discontinuous map changes, and may
159 /// be a small error from smoothly-varying maps that are not sufficiently
160 /// resolved from one element to the next.

296 template <size_t VolumeDim, typename... Tags>
297 void Weno<VolumeDim, tmpl::list<Tags...>>::package_data(
298  const gsl::not_null<PackagedData*> packaged_data,
299  const typename Tags::type&... tensors, const Mesh<VolumeDim>& mesh,
300  const std::array<double, VolumeDim>& element_size,
301  const OrientationMap<VolumeDim>& orientation_map) const noexcept {
302  // By always initializing the PackagedData Variables member, we avoid an
303  // assertion that arises from having a default-constructed Variables in a
304  // disabled limiter. There is a performance cost, because the package_data()
305  // function does non-zero work even for a disabled limiter... but since the
306  // limiter should never be disabled in a production simulation, this cost
307  // should never matter.
308  (packaged_data->volume_data).initialize(mesh.number_of_grid_points());
309 
310  if (UNLIKELY(disable_for_debugging_)) {
311  // Do not initialize packaged_data
312  // (except for the Variables member "volume_data", see above)
313  return;
314  }
315 
316  const auto wrap_compute_means = [&mesh, &packaged_data](
317  auto tag, const auto tensor) noexcept {
318  for (size_t i = 0; i < tensor.size(); ++i) {
319  // Compute the mean using the local orientation of the tensor and mesh.
320  get<::Tags::Mean<decltype(tag)>>(packaged_data->means)[i] =
321  mean_value(tensor[i], mesh);
322  }
323  return '0';
324  };
325  expand_pack(wrap_compute_means(Tags{}, tensors)...);
326 
327  packaged_data->element_size =
328  orientation_map.permute_from_neighbor(element_size);
329 
330  const auto wrap_copy_tensor = [&packaged_data](auto tag,
331  const auto tensor) noexcept {
332  get<decltype(tag)>(packaged_data->volume_data) = tensor;
333  return '0';
334  };
335  expand_pack(wrap_copy_tensor(Tags{}, tensors)...);
336  packaged_data->volume_data = orient_variables(
337  packaged_data->volume_data, mesh.extents(), orientation_map);
338 
339  packaged_data->mesh = orientation_map(mesh);
340 }

342 template <size_t VolumeDim, typename... Tags>
343 bool Weno<VolumeDim, tmpl::list<Tags...>>::operator()(
344  const gsl::not_null<std::add_pointer_t<typename Tags::type>>... tensors,
345  const Mesh<VolumeDim>& mesh, const Element<VolumeDim>& element,
346  const std::array<double, VolumeDim>& element_size,
347  const std::unordered_map<
348  std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>, PackagedData,
349  boost::hash<std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>>>&
350  neighbor_data) const noexcept {
351  if (UNLIKELY(disable_for_debugging_)) {
352  // Do not modify input tensors
353  return false;
354  }
355 
356  // Check that basis is LGL or LG
357  // A Legendre basis is assumed for the oscillation indicator (used in both
358  // SimpleWeno and Hweno) and in the Hweno reconstruction.
359  ASSERT(mesh.basis() == make_array<VolumeDim>(Spectral::Basis::Legendre),
360  "Unsupported basis: " << mesh);
361  ASSERT(mesh.quadrature() ==
362  make_array<VolumeDim>(Spectral::Quadrature::GaussLobatto) or
363  mesh.quadrature() ==
364  make_array<VolumeDim>(Spectral::Quadrature::Gauss),
365  "Unsupported quadrature: " << mesh);
366 
367  // Enforce restrictions on h-refinement, p-refinement
368  if (UNLIKELY(alg::any_of(element.neighbors(),
369  [](const auto& direction_neighbors) noexcept {
370  return direction_neighbors.second.size() != 1;
371  }))) {
372  ERROR("The Weno limiter does not yet support h-refinement");
373  // Removing this limitation will require:
374  // - Generalizing the computation of the modified neighbor solutions.
375  // - Generalizing the WENO weighted sum for multiple neighbors in each
376  // direction.
377  }
378  alg::for_each(neighbor_data, [&mesh](const auto& neighbor_and_data) noexcept {
379  if (UNLIKELY(neighbor_and_data.second.mesh != mesh)) {
380  ERROR("The Weno limiter does not yet support p-refinement");
381  // Removing this limitation will require generalizing the
382  // computation of the modified neighbor solutions.
383  }
384  });
385 
386  if (weno_type_ == WenoType::Hweno) {
387  // Troubled-cell detection for HWENO flags the element for limiting if any
388  // component of any tensor needs limiting.
389  const bool cell_is_troubled =
390  Tci::tvb_minmod_indicator<VolumeDim, PackagedData, Tags...>(
391  tvb_constant_, (*tensors)..., mesh, element, element_size,
392  neighbor_data);
393  if (not cell_is_troubled) {
394  // No limiting is needed
395  return false;
396  }
397 
398  std::unordered_map<
399  std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>, DataVector,
400  boost::hash<std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>>>
401  modified_neighbor_solution_buffer{};
402  for (const auto& neighbor_and_data : neighbor_data) {
403  const auto& neighbor = neighbor_and_data.first;
404  modified_neighbor_solution_buffer.insert(
405  make_pair(neighbor, DataVector(mesh.number_of_grid_points())));
406  }
407 
408  EXPAND_PACK_LEFT_TO_RIGHT(Weno_detail::hweno_impl<Tags>(
409  make_not_null(&modified_neighbor_solution_buffer), tensors,
410  neighbor_linear_weight_, mesh, element, neighbor_data));
411  return true; // cell_is_troubled
412 
413  } else if (weno_type_ == WenoType::SimpleWeno) {
414  // Buffers and pre-computations for TCI
415  Minmod_detail::BufferWrapper<VolumeDim> tci_buffer(mesh);
416  const auto effective_neighbor_sizes =
417  Minmod_detail::compute_effective_neighbor_sizes(element, neighbor_data);
418 
419  // Buffers for simple WENO implementation
420  std::unordered_map<
421  std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>,
422  intrp::RegularGrid<VolumeDim>,
423  boost::hash<std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>>>
424  interpolator_buffer{};
425  std::unordered_map<
426  std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>, DataVector,
427  boost::hash<std::pair<Direction<VolumeDim>, ElementId<VolumeDim>>>>
428  modified_neighbor_solution_buffer{};
429 
430  bool some_component_was_limited = false;
431 
432  const auto wrap_minmod_tci_and_simple_weno_impl =
433  [this, &some_component_was_limited, &tci_buffer, &interpolator_buffer,
434  &modified_neighbor_solution_buffer, &mesh, &element, &element_size,
435  &neighbor_data,
436  &effective_neighbor_sizes](auto tag, const auto tensor) noexcept {
437  for (size_t tensor_storage_index = 0;
438  tensor_storage_index < tensor->size(); ++tensor_storage_index) {
439  // Check TCI
440  const auto effective_neighbor_means =
441  Minmod_detail::compute_effective_neighbor_means<decltype(tag)>(
442  tensor_storage_index, element, neighbor_data);
443  const bool component_needs_limiting = Tci::tvb_minmod_indicator(
444  make_not_null(&tci_buffer), tvb_constant_,
445  (*tensor)[tensor_storage_index], mesh, element, element_size,
446  effective_neighbor_means, effective_neighbor_sizes);
447 
448  if (component_needs_limiting) {
449  if (modified_neighbor_solution_buffer.empty()) {
450  // Allocate the neighbor solution buffers only if the limiter is
451  // triggered. This reduces allocation when no limiting occurs.
452  for (const auto& neighbor_and_data : neighbor_data) {
453  const auto& neighbor = neighbor_and_data.first;
454  modified_neighbor_solution_buffer.insert(make_pair(
455  neighbor, DataVector(mesh.number_of_grid_points())));
456  }
457  }
458  Weno_detail::simple_weno_impl<decltype(tag)>(
459  make_not_null(&interpolator_buffer),
460  make_not_null(&modified_neighbor_solution_buffer), tensor,
461  neighbor_linear_weight_, tensor_storage_index, mesh, element,
462  neighbor_data);
463  some_component_was_limited = true;
464  }
465  }
466  return '0';
467  };
468  expand_pack(wrap_minmod_tci_and_simple_weno_impl(Tags{}, tensors)...);
469  return some_component_was_limited; // cell_is_troubled
470  } else {
471  ERROR("WENO limiter not implemented for WenoType: " << weno_type_);
472  }
473 
474  return false; // cell_is_troubled
475 }
